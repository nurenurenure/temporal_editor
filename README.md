## 🎯 Постановка задачи и актуальность
Платформа оркестрации Temporal предоставляет мощный инструмент для управления распределенными транзакциями, однако создание процессов требует написания сложного кода на Go/TypeScript. Данный проект решает проблему высокого порога входа (No-Code/Low-Code подход), позволяя бизнес-аналитикам и разработчикам визуально проектировать, валидировать и разворачивать отказоустойчивые workflow без написания кода, транслируя графические узлы в декларативный DSL-формат.

Визуальный веб-редактор и оркестратор долгоживущих бизнес-процессов (Workflows) для платформы **Temporal**, работающий на базе декларативного движка **temporal-dsl** (Zigflow).

Проект предоставляет полноценный UI/UX интерфейс, который позволяет проектировать сложные распределенные сценарии в виде интерактивного графа, автоматически транслировать их в валидный YAML-конфигурацию движка `temporal-dsl`, а также запускать, отслеживать состояние и просматривать пошаговую историю событий в реальном времени.

**Репозиторий проекта:** [github.com/nurenurenure/temporal_editor](https://github.com/nurenurenure/temporal_editor)

## 🏗 Архитектура системы

Проект спроектирован по принципу микросервисной архитектуры и разворачивается в изолированной среде:

```
[ Frontend: React / ReactFlow ] --(Прокси: Nginx /api/*)--> [ Backend: Go REST API ]
                                                                   |
                                          +------------------------+------------------------+
                                          |                                                 |
                                 [ PostgreSQL 15 ]                              [ Docker Socket (dind) ]
                           (Хранение метаданных & JSON)                        (Перезапуск Zigflow Worker)
                                                                                            |
                                                                                  [ zigflow-worker ]
                                                                             (Исполнение .yaml в Temporal)
```

1. **Frontend (React + Vite + TypeScript + ReactFlow):** Одностраничное приложение (SPA), упакованное в высокопроизводительный production-сборщик под управлением Nginx. Nginx выступает единой точкой входа (Reverse Proxy), раздает статику на порту `80` и незаметно для пользователя перенаправляет запросы `/api/*` на бэкенд.
2. **Backend (Go REST API):** Реализует CRUD для управления процессами, валидацию графов, генерацию совместимых с DSL YAML-файлов и прямое взаимодействие с Temporal оркестратором через официальный Go SDK. Содержит встроенный Docker-клиент для динамического управления контейнером воркера через системный сокет (`/var/run/docker.sock`).
3. **Хранилище (PostgreSQL 15):** Единая база данных, в которой сохраняются как метаданные воркфлоу для бэкенда, так и системные таблицы самого ядра Temporal. Данные защищены именованным томом `postgres_data`.
4. **Zigflow Worker (`ghcr.io/zigflow/zigflow`):** Изолированный контейнер, к которому примонтирована директория с динамически генерируемыми YAML-файлами (`/app/workflows/`). При изменении схемы воркфлоу Go-сервер автоматически обновляет файл и перезапускает контейнер воркера для мгновенного применения изменений.

## 🗺 API Эндпоинты (REST API)

### Управление Воркфлоу (CRUD)
Все обработчики принимают и возвращают структуры данных в формате JSON.

| Метод | Путь | Описание |
| :--- | :--- | :--- |
| `GET` | `/api/workflows` | Получить список всех сохраненных воркфлоу (сортировка DESC по дате создания) |
| `POST` | `/api/workflows` | Создать новый воркфлоу (вызывает валидацию шагов) |
| `GET` | `/api/workflows/{id}` | Получить детальную конфигурацию воркфлоу по его UUID |
| `PUT` | `/api/workflows/{id}` | Обновить существующий воркфлоу (перезаписывает БД и YAML, перезапускает воркер) |
| `DELETE` | `/api/workflows/{id}` | Удалить воркфлоу (каскадно удаляет запись из БД и файл конфигурации с диска) |

### Запуск и Мониторинг Выполнения

* **`POST /api/workflows/{id}/run`**
  Запускает выполнение конкретного бизнес-процесса в Temporal. Принимает необязательный JSON-объект входных данных (`data`). Возвращает: `workflow_id`, `run_id` и прямую ссылку на Temporal UI.
* **`GET /api/runs/{workflowId}/{runId}`**
  Получает низкоуровневые детали запущенной сессии напрямую из Temporal оркестратора с использованием метода `DescribeWorkflowExecution` (статус выполнения, время старта/завершения).
* **`GET /api/runs/{workflowId}/{runId}/history`**
  Возвращает массив произошедших событий с их ID, типом и временными метками для отображения интерактивного лога выполнения.

## 🗄 Структура Базы Данных

Таблица `workflows` в PostgreSQL имеет следующую схему:
* `id` (`UUID`, PRIMARY KEY, автогенерация)
* `name` (`VARCHAR(255)`, NOT NULL)
* `description` (`TEXT`)
* `config_json` (`JSONB`, хранит упорядоченный массив шагов и координат UI-нод графа)
* `created_at` (`TIMESTAMPTZ`, дефолтное значение текущего времени)

## 🎨 Поддерживаемые типы шагов (DSL Ноды)

В визуальном редакторе доступны следующие блоки для построения бизнес-логики:
1. **set (Присвоение):** Инициализация или модификация переменных контекста (ключ-значение).
2. **wait (Ожидание):** Приостановка процесса на заданное количество секунд (использует нативные таймеры Temporal).
3. **call_http:** Выполнение внешнего HTTP-запроса с выбором метода (GET, POST и т.д.) и URL-адреса.
4. **call_activity:** Вызов классической Activity, зарегистрированной на стороне воркера (код на Go/TS).
5. **call_grpc:** Вызов метода микросервиса по gRPC с указанием хоста, сервиса, метода и payload.
6. **listen (Ожидание сигнала):** Остановка процесса до тех пор, пока извне не придет Temporal Signal с определенным ID.
7. **export (Экспорт сигнала):** Передача данных наружу или обновление контекста на основе принятого сигнала.
8. **switch (Ветвление):** Условный оператор. На основе перечня условий (`when`) направляет поток в соответствующую именованную ветку (`then`).
9. **for (Цикл по массиву):** Итерация по элементам массива с выполнением вложенной цепочки шагов `do`.
10. **parallel:** Разветвление логики на несколько параллельно выполняющихся цепочек шагов с обязательным последующим слиянием в узле **join**.
11. **tryCatch:** Конструкция обработки исключений с выделенными визуальными выходами для основного потока (`try`) и блока обработки ошибок (`catch`).

## 🛡 Валидация графов

Перед деплоем (`Deploy`) или запуском (`Run`), на фронтенде и бэкенде срабатывает комплексный движок валидации, блокирующий сохранение некорректных схем:
* **Проверка связности:** В графе должен быть строго один стартовый узел (без входящих ребер). Все остальные узлы должны быть достижимы из стартового. Несвязные изолированные блоки запрещены.
* **Контроль зацикливания:** Алгоритм поиска в глубину (DFS) с раскраской вершин проверяет граф на отсутствие циклических зависимостей, которые могли бы привести к вечной блокировке воркера.
* **Уникальность имен:** Запрещено дублирование названий шагов (ключей) внутри одного бизнес-процесса.
* **Специфические правила блоков:** Блок `parallel` обязан иметь не менее 2 исходящих ребер; блок `tryCatch` должен содержать обе настроенные ветки; линейные ноды (типа `set` или `call_http`) не могут разветвляться и иметь более одного исходящего ребра.

## 🚀 Быстрый запуск

### Требования
* Установленный **Docker** и плагин **Docker Compose**.
* Доступ к интернету (для загрузки базовых образов и библиотек).

### Инструкция по запуску всей инфраструктуры в 1 команду:

1. Склонируйте репозиторий:
   ```bash
   git clone https://github.com/nurenurenure/temporal_editor.git
   cd temporal_editor
   ```

2. Запустите оркестрацию контейнеров:
   ```bash
   docker compose up --build -d
   ```

3. Docker Compose автоматически настроит цепочку зависимостей (Healthchecks):
   * Сначала поднимется база данных PostgreSQL 15.
   * После готовности БД запустится ядро Temporal Server и выполнит необходимые миграции.
   * Затем одновременно стартуют Temporal Web UI, Go-Backend (который автоматически развернет дочерний контейнер `zigflow-worker`) и контейнер Nginx с React-фронтендом.

### Доступные веб-интерфейсы:
* **Основное приложение (Редактор воркфлоу):** [http://localhost](http://localhost) (порт `80`)
* **Официальная панель управления Temporal Web UI:** [http://localhost:8233](http://localhost:8233) (мониторинг низкоуровневых стейтов durable execution)

## 🔮 Направления дальнейшего развития (Масштабирование)

Текущая версия проекта представляет собой полнофункциональный MVP. В качестве дальнейшего расширения системы в рамках научно-исследовательской работы (НИР) или выпускной квалификационной работы (ВКР) планируются следующие улучшения:

### 1. Версионирование воркфлоу (Workflow Versioning)
* **Проблема:** Сейчас при изменении графа старый YAML-файл перезаписывается. Если в этот момент выполнялись долгоживущие процессы (которые могут висеть днями или неделями), они завершатся с ошибкой из-за несовместимости схем (Non-Deterministic Error).
* **Решение:** Интеграция нативного механизма версионирования Temporal (Workflow Patching / Versioning API). Сохранение истории версий в БД PostgreSQL и генерация уникальных YAML для каждой версии схемы (`workflow_v1.yaml`, `workflow_v2.yaml`), чтобы старые процессы добегали по старой логике, а новые запускались по новой.

### 2. Реализация распределенного пула воркеров
* **Проблема:** Текущая архитектура использует один локальный контейнер `zigflow-worker`, управляемый через Docker Socket бэкенда, что создает Single Point of Failure (единую точку отказа) и ограничивает производительность.
* **Решение:** Переход от Docker API к оркестрации через **Kubernetes (K8s) API**. Бэкенд будет динамически масштабировать реплики воркеров (Horizontal Pod Autoscaler) в зависимости от нагрузки на очередь задач (Task Queue) в Temporal.

### 3. Комплексный мониторинг и алертинг
* Реализация дашборда со сквозной аналитикой: среднее время выполнения процессов, процент падения кастомных HTTP/gRPC шагов.
* Интеграция с Prometheus и Grafana для сбора бизнес-метрик воркфлоу и отправка уведомлений в Telegram/Slack при критическом падении бизнес-процесса (например, если блок `tryCatch` ушел в глухую ошибку).
